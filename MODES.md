# Operational Modes

Log-Sense uses a Tri-Mode architecture to provide versatile security analysis environments. Each mode is strictly isolated at the database and API level.

## 🟢 Simulation Mode (Interactive Sandbox)
**Purpose**: Synthetically test SOC response and rule engine behavior.
- **Inbound Data**: Generated by `backend/simulator.js`.
- **Enabled Features**:
  - Full Attack Simulator (Brute Force, DDoS, Port Scan).
  - Real-time Log Streaming.
  - Anomaly detection on synthetic traffic.
- **State Management**: Uses Socket.IO to push live updates to the frontend.

## 🔴 AWS Live Mode (Production Monitoring)
**Purpose**: Monitor remote Linux servers in real-time.
- **Inbound Data**: Streamed via SSH2 from a remote `auth.log` or `syslog` file.
- **Enabled Features**:
  - Live EC2 connection management.
  - Real-world rule matching.
  - Attacker IP blocking (Logic only).
- **Disabled Features**:
  - Attack Simulator (Blocked to prevent accidental production impact).

## 🟡 Forensic Mode (Offline Investigation)
**Purpose**: Post-mortem analysis of security events from static log files.
- **Inbound Data**: Provided via user upload (`POST /api/forensic/upload`).
- **Enabled Features**:
  - **AI-Assisted Intelligent Log Parsing**: Uses Gemini AI to dynamically extract entities (IP, User, Event) from unstructured bulk text.
  - **Static Security Intelligence Scan**: Immediate application of the rule engine to the uploaded dataset.
  - **Contextual Risk Enrichment**: AI explanations generated for every parsed event.
- **Disabled/Modified Features**:
  - Attack Simulation barred.
  - Real-time streaming is simulated (log-ingestion events emitted on upload).
- **Behavior**: To ensure investigation integrity, every new upload initiates a **Hard Reset** of global forensic state, clearing previous logs, alerts, and incidents.

---

## Technical Isolation
All data entities (`logs`, `alerts`, `incidents`) contain a `source` field:
- `SIMULATION`
- `AWS`
- `FORENSIC`

The backend middleware `resolveSource()` ensures that API requests only return data corresponding to the user's active mode selection, preventing cross-mode data leakage.
